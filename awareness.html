Why doesn’t Bluebook automatically ban you?

Because of this function (functions renamed for clarity):

Bluebook Assessment Reporter 
- (void)persistAssessmentReport:(NSDictionary *)report 
                     forSession:(NSString *)sessionID 
                   withSeverity:(AEAssessmentSeverity)severity {
    
    // Encrypts report + attaches device fingerprint + SIP status
    [self.secureStorage queueReport:report 
                         sessionID:sessionID 
                        severity:severity 
                        requiresHumanReview:YES];
    
    // Triggers 5-stage manual review pipeline
    [self notifyReviewQueueWithReportID:report.reportID];
}

These events are logged and sent for manual review. There are currently 5 stages of review. This is why some scores get delayed for weeks.

How a typical “stealthy” injection bypass works on macOS (and why it fails)

A normal bypass workflow looks like this:

  1. Build a .dylib with the bypass logic
  2. Use dlopen() or DYLD_INSERT_LIBRARIES to inject into Bluebook
  3. Disable SIP so the injection can actually work

When SIP is off, Apple removes most sandboxing protections. This is why almost every seller tells you to turn SIP off.
